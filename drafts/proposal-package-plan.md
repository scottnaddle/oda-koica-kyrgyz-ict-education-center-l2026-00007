# Proposal Package Plan

> Status: internal planning draft
> Opportunity: L2026-00007-1
> Purpose: 평가축 기준으로 proposal 제출 패키지를 어떻게 분해할지 설계

## Evaluation-axis assumption
현재 확보된 자료와 실제 우수 제안서 샘플 패턴을 기준으로 보면,
이 건은 단일 연속 본문보다 아래 평가축으로 나눠 생각하는 것이 유리하다.

- company introduction
- technical proposal
- business/project management
- project support
- value-added / additional proposal

다만 실제 발주 형식이 완전 분리 제출을 요구하는지 여부는 추가 확인이 필요하다.
현 단계에서는 **내부 작성/준비 구조를 위 패키지 기준으로 나눈다**는 의미로 본다.

---

## Overall recommendation
이 사업은 전략 시그널은 강하지만 직접 수행 적합도는 조건부이므로,
proposal package도 “우리가 다 할 수 있다”는 과장형이 아니라 아래 원칙으로 구성하는 것이 맞다.

1. **기술 부문**에서는 정책/전략성 + ICT 교육센터 맥락을 강하게 잡는다.
2. **사업관리 부문**에서는 파트너링 전제와 역할 분담의 현실성을 보여준다.
3. **프로젝트 지원 부문**에서는 QA/보고/운영 readiness를 구조적으로 제시한다.
4. **추가제안 부문**에서는 과도한 확장보다 파트너링/운영/교육 연계 가치를 차별화 포인트로 둔다.

---

## Recommended package split

### 1. Company Introduction Package
#### Purpose
- 왜 유비온이 이 사업을 볼 가치가 있는 기관인지 설명
- 완전한 설계사 정체성 대신, 교육/디지털/ODA 맥락 연결 능력을 증명

#### Key sections
- 회사 개요
- 교육/에듀테크/디지털 전환 관련 정체성
- ODA/국제협력/교육 인프라 관련 실적
- 유사 사업군과의 연결 포인트
- 본 사업에서 유비온이 제공할 수 있는 고유 가치

#### Evidence to include
- 유비온 IR/회사 소개 자료
- ODA/교육/디지털 전환 실적
- ICT 교육센터와 연결 가능한 레퍼런스

#### Writing caution
- 건축설계 주사업자처럼 보이게 과장하지 말 것
- 기관 신뢰도와 전략 맥락 연결을 중심으로 쓸 것

---

### 2. Technical Proposal Package
#### Purpose
- 이 사업의 정책적 의미와 사업 필요성을 정확히 읽고 있다는 점을 보여줌
- ICT 교육센터 사업의 전략적 의미를 제시

#### Key sections
- 사업 배경 및 필요성
- Kyrgyz 디지털 전환 / 국가 정보화 역량 강화 맥락
- ICT 교육센터의 기능적/전략적 의미
- 과업 범위 해석
- 설계·입찰지원·author’s supervision 범위 이해
- 기대효과 및 사업 성공 조건

#### Evidence to include
- RFP / ToR / BDS 근거
- 정책 부합성 근거
- 기존 full proposal body draft의 정책/배경 부분
- scope/deliverables summary

#### Writing caution
- 기술 부문은 화려한 솔루션 제안보다 **정확한 문제정의와 범위 이해**가 우선
- 실제 범위가 설계 consultant service임을 흐리지 말 것

---

### 3. Business / Project Management Package
#### Purpose
- 실제 수행 구조가 어떻게 굴러갈지에 대한 신뢰를 제공
- 현지화, 인력, 역할 분담, 파트너링 필요성을 보여줌

#### Key sections
- 투입인력 구조
- PM / 핵심 인력 역할
- 파트너링 가설 및 역할 분담
- 현지화 전략
- 보고체계 / 협의체계
- 리스크 대응 및 의사결정 구조

#### Evidence to include
- qualification/compliance summary
- proposal kickoff의 partner/resource gaps
- 향후 파트너 구조 메모

#### Writing caution
- JV 불가와 local design license requirement를 무시한 조직도를 그리면 안 됨
- `유비온 + 현지 적격/설계 파트너` 구조 전제가 자연스럽게 드러나야 함

---

### 4. Project Support Package
#### Purpose
- 사업을 안정적으로 굴릴 수 있는 운영 준비도를 보여줌
- 기술 본문과 별도로 QA/운영/지원 readiness를 증명

#### Key sections
- 품질보증 전략
- 산출물 검토 체계
- 보고 주기 및 승인 흐름
- 문서/커뮤니케이션 관리
- 리스크/이슈 관리
- 필요 시 교육훈련/운영지원/유지관리 관점의 확장 메모

#### Evidence to include
- 기존 제안서 분할 샘플의 프로젝트 지원 패턴
- ODA 운영형 문서 구조
- proposal prep checklist

#### Writing caution
- 실제 발주범위를 넘어서는 유지관리 약속은 과도하게 쓰지 말 것
- 설계/감리 consultant 범위 안에서 현실적인 support readiness를 보여줄 것

---

### 5. Value-added / Additional Proposal Note
#### Purpose
- 기본 제안 외에 경쟁우위를 만드는 추가 포인트 제시
- 단, 본 사업 특성상 무리한 확장보다 현실적 보강안을 제안

#### Key sections
- 추가 제안 요약
- 추가 자문단 / advisory support
- 디지털 교육 운영 관점 확장 제안
- 향후 ICT 교육센터 활용성 제고 아이디어
- 왜 이것이 차별화 포인트인지

#### Evidence to include
- split proposal sample의 추가제안 패턴
- 유비온의 교육/디지털 운영 강점

#### Writing caution
- 추가제안이 본 범위를 압도하면 안 됨
- `설계 용역에 덧붙는 전략적 가치` 수준으로 절제할 것

---

## Suggested writing order
실무 작성 순서는 아래가 좋다.

1. technical proposal package
2. business/project management package
3. project support package
4. company introduction package
5. value-added proposal note

이 순서가 좋은 이유:
- 먼저 사업을 정확히 읽고
- 그 다음 누가 어떻게 할지 쓰고
- 그 뒤에 지원체계를 붙이고
- 마지막에 회사 신뢰도와 추가차별화를 올리는 편이 논리적으로 안정적이다.

## Recommendation for this opportunity
현재 이 건에서 가장 먼저 써볼 가치가 큰 건 아래 두 개다.

1. **Technical Proposal Package**
2. **Business / Project Management Package**

이유:
- 이 사업의 승패는 결국 `우리가 사업을 얼마나 정확히 이해하느냐`와
  `현실적인 수행 구조를 얼마나 설득력 있게 보여주느냐`에 달려 있기 때문이다.

즉, 지금 단계에서 가장 중요한 것은
**좋아 보이는 추가제안보다도, 기술 이해 + 현실적인 수행체계**다.
